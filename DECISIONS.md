# DECISIONS.md

작업 중 방향을 바꾸거나 중요한 선택을 한 시점의 기록.
"왜 이렇게 했는지"를 남겨 나중에 같은 고민을 반복하지 않기 위한 파일.

---

## 2026-04-23

### DB 연결 방식 — SQLAlchemy 대신 asyncpg 직접 사용

**결정:** `asyncpg.create_pool()` 직접 사용  
**이유:** 4인 소규모 팀, 단순한 쿼리 위주. SQLAlchemy의 ORM/마이그레이션 복잡도가 현 단계에서 불필요. asyncpg로 경량하게 유지.  
**재검토 시점:** 쿼리가 복잡해지거나 마이그레이션 관리가 필요해질 때.

---

### 파서 미완료 기본값 정책

**결정:** 완료/미완료 표기가 없으면 → 미완료로 처리  
**이유:** EEEE/Natae는 "표기 없음 = 완료" 스타일이지만, 업무계획과 업무보고를 구분하는 유일한 기준이 표기이기 때문에 기본값을 미완료로 설정. 신뢰도 0% → Phase 2 Claude 폴백으로 보완 예정.

---

### 무지 Notion 링크 보고 방식

**결정:** 현재 파싱 대상에서 제외, 추후 운영 규칙으로 해결  
**이유:** 무지는 Notion 문서 링크만 채널에 올리는 경우가 있어 텍스트 파싱 불가. 봇 운영 시작 시 "채널에 텍스트로 직접 작성" 규칙 공지 예정.

---

### 이미지 메시지 대응

**결정:** 이미지 파싱 구현하지 않음, 텍스트 전용 운영 규칙 적용  
**이유:** OCR 연동은 비용/복잡도 대비 효과 낮음. 봇 도입 시 텍스트 전용 규칙을 팀에 공지하는 것으로 대체.

## 2026-04-24

### Phase 1 작업 순서 변경 — 채널톡 연동보다 파서 먼저

**결정:** 1-3(채널톡 연동) → 1-4(파서) 순서를 바꿔 파서를 먼저 구현  
**이유:** 채널톡 워크스페이스 관리자 권한 확보가 지연됨. 파서는 채널톡 없이 독립적으로 구현 및 테스트 가능. 실제 메시지 데이터(`sample_messages.txt`)로 검증까지 완료.

---

### 파서 패턴 도출 방식 — 실데이터 기반

**결정:** 가상의 샘플 대신 실제 채널톡 대화 내역을 `sample_messages.txt`로 저장해 패턴 분석  
**이유:** 멤버별 완료 표기가 매우 다양함 (O/ㅇ/세모/(미완료)/(o)/수치기록 등). 실데이터 없이 규칙을 작성하면 실제 운영에서 오파싱 비율이 높아질 것으로 판단.  
**결과:** 208건 파싱 성공 (92%), 실패 18건은 모두 내용 없는 메시지로 정상 실패.

---

### DATABASE_URL 형식 수정

**결정:** `postgresql+asyncpg://` → `postgresql://`  
**이유:** `postgresql+asyncpg://`는 SQLAlchemy의 드라이버 접두어. asyncpg 직접 사용 시에는 `postgresql://`만 인식함. `.env.example` 및 `database.py`에서 자동 변환 로직 추가.
